--------------------------------------------------------------------------------
ISSUE 1 — 422: "GitHub Actions is not permitted to approve pull requests"
--------------------------------------------------------------------------------

Root cause: This is a GitHub *repository* setting, not a code bug (as you
already suspected).

  ai-pr-reviewer-test repo
    -> Settings -> Actions -> General -> "Workflow permissions"
    -> enable "Allow GitHub Actions to create and approve pull requests"

The bot is allowed to POST review comments and issue comments, but the default
GITHUB_TOKEN is forbidden from submitting an APPROVE review. That is exactly the
error PyGithub surfaces from POST {pr}/reviews with event="APPROVE".

Where it happens in the code (for reference, do NOT need to change):
  prlens/github/pr_publisher.py:206  submit_review() -> create_review(event="APPROVE")
  called from prlens/core/agent.py:25

SECONDARY CODE OBSERVATION (not the cause, but explains the symptom ordering):

  agent.run() executes in this order (agent.py:22-26):
      1. apply_labels
      2. post_summary          <-- the "Score 100 / Approved" comment is posted HERE
      3. post_inline_comments
      4. submit_review         <-- THIS raises the uncaught 422, killing the job
      5. assign_reviewers      <-- never runs

  That is why you SEE the summary comment ("✅ Approved, Score 100") but the
  check still shows "Failing after 27s": the summary is posted first, then
  submit_review() crashes.

  Unlike assign_reviewers() (pr_publisher.py:278-284, which wraps its API calls
  in try/except GithubException), submit_review() does NOT catch GithubException.
  So the 422 propagates out of agent.run() -> main.py -> process exits non-zero
  -> the Action is marked failed.

  Enabling the repo setting removes the 422 for the APPROVE path. If you also
  want the job to survive a bad token/permission state gracefully instead of
  hard-failing, submit_review() is the place that would need the same
  try/except treatment assign_reviewers() already has. (Left as a note only —
  no change made per your instruction.)


--------------------------------------------------------------------------------
ISSUE 2 — Score 100, zero comments/positives/recommendations, "Approved"
          on code containing hardcoded secrets + SQL injection
--------------------------------------------------------------------------------

This one IS a real code defect. Two things had to line up to produce the exact
symptom you saw (everything "None", score 100, Approved, and INCOMPLETE NOT
triggered). Walk the pipeline:

STEP 0 — Rule out the file filter (your first hypothesis):
  prlens/config/settings.py:43 filter_files()
  The bad file is `.py`. EXTENSION_TO_LANGUAGE[".py"] = SupportedLanguages.PYTHON,
  and target_languages in .aireviewer.yml includes "python". So the file PASSES
  the filter. total_files = 1. The filter is NOT the cause here. Good — that
  rules out the "wrong language / filtered out" theory.

STEP 1 — Where the empty result actually comes from (THE BUG):

  prlens/llm/analyzer.py:82-86
      def _parse_response(self, response: str) -> FileReviewResponse:
          response = response.strip()
          data = json.loads(response)
          return FileReviewResponse(**data)

  prlens/models/review.py:20-23
      class FileReviewResponse(BaseModel):
          comments:        list[ReviewComment] = Field(default_factory=list)
          positives:       list[str]           = Field(default_factory=list)
          recommendations: list[str]           = Field(default_factory=list)

  KEY FACT: FileReviewResponse has NO model_config, so it uses Pydantic v2's
  DEFAULT behavior: extra="ignore". Combined with every field having a default
  (default_factory=list), this means:

      * If the LLM returns a valid JSON OBJECT whose keys do NOT match
        {comments, positives, recommendations} — e.g. it wraps the payload
        ({"review": {...}}), renames a key ({"issues": [...]}), or returns a
        differently-shaped object — then:
            - json.loads() SUCCEEDS (it is valid JSON)
            - FileReviewResponse(**data) SUCCEEDS: the unknown keys are silently
              IGNORED and the missing "comments"/"positives"/"recommendations"
              keys fall back to EMPTY LISTS.
            - NO exception is raised.

  This is the M8 regression. After switching to response_format=
  {"type": "json_object"} and simplifying _parse_response to a bare
  json.loads + FileReviewResponse(**data), there is no longer any check that the
  parsed object actually CONTAINS the expected keys. A structurally-valid but
  schema-mismatched response degrades SILENTLY into an all-empty review.

STEP 2 — Why INCOMPLETE was NOT triggered (this is the giveaway):

  prlens/llm/analyzer.py:50-62
      Only an EXCEPTION from analyze_file() adds a file to failed_files, which is
      what drives INCOMPLETE/TOTAL_FAILURE (pr_publisher.py:44-49).

  Because the silent-empty path above raises NO exception, failed_files stays
  empty. So:
      - not TOTAL_FAILURE (failed_files != total_files)
      - not INCOMPLETE   (failed_files is empty)
  Exactly matching your observation: "INCOMPLETE wasn't triggered."

  Contrast: the parse failures that WOULD have been caught (and shown as
  INCOMPLETE) are only:
      - json.loads() raising (e.g. markdown fences, truncated JSON), or
      - a top-level JSON *array* -> FileReviewResponse(**list) -> TypeError, or
      - present-but-invalid values (bad enum, comments:null) -> ValidationError.
  The one case that slips through silently is the one that bit you: a valid JSON
  *dict* with missing/renamed keys.

STEP 3 — Empty comments -> vacuous "Approved" (the amplifier):

  prlens/llm/analyzer.py:88-101  _calculate_score()
      score = 100 - sum(penalties)   ->  no comments == no penalties == 100

  prlens/github/pr_publisher.py:44-57  _determine_review_outcome()
      failed_files empty, has_critical_issues False, score 100 > approve_threshold(80)
      ->  ReviewOutcome.APPROVED

  There is NO guard for "nothing was actually found." A review that produced
  zero comments AND zero positives AND zero recommendations is indistinguishable,
  to the outcome logic, from a genuinely clean PR — both auto-APPROVE. That is
  how deliberately-bad code sails through with 100/100.

  (Then submit_review() tries a real APPROVE -> Issue 1's 422 -> job fails, which
   is the only reason you noticed at all.)

--------------------------------------------------------------------------------
ROOT-CAUSE SUMMARY (Issue 2)
--------------------------------------------------------------------------------

Primary:  _parse_response() (analyzer.py:82-86) + FileReviewResponse's implicit
          extra="ignore" (review.py:20-23) let a valid-JSON-but-wrong-shape LLM
          response collapse into an all-empty result with NO error. This is the
          M8 change removing the previous robustness.

Amplifier: The scoring + outcome logic treats "no findings" as a perfect,
          approvable PR. There is no "low-signal / suspicious empty response"
          safeguard, so a degenerate empty parse becomes a confident APPROVE.

Why it looked like "the LLM returned an empty comments array": functionally it
was empty, but the emptiness most likely came from the SILENT PARSE FALLBACK,
not from gpt-4o genuinely judging hardcoded secrets + SQL injection as clean.

HOW TO CONFIRM (no code changes required)
--------------------------------------------------------------------------------

1. In the Actions run log, look for the raw model output. With
   logging.basicConfig(level=INFO) (main.py:13) the response body is NOT logged
   today, so temporarily (locally) print `response` inside _parse_response, or
   re-run the model on the same diff and inspect the JSON. Check whether the
   top-level keys are literally "comments"/"positives"/"recommendations".

2. Reproduce in isolation:
       from prlens.models.review import FileReviewResponse
       FileReviewResponse(**{"issues": [1,2,3]})   # wrong key
       # -> FileReviewResponse(comments=[], positives=[], recommendations=[])
       #    NO error. This demonstrates the silent-empty fallback.

3. Confirm there was NO "Failed to analyze <file>" warning in the log
   (analyzer.py:62). Its absence + empty result == the silent-parse path, not an
   exception path.

--------------------------------------------------------------------------------
SUGGESTED FIX DIRECTIONS (for later — NOT applied here)
--------------------------------------------------------------------------------

A. Make the parse strict so a schema mismatch FAILS LOUDLY (becomes INCOMPLETE
   instead of a silent approve):
     - add model_config = ConfigDict(extra="forbid") to FileReviewResponse, and/or
     - in _parse_response, assert the expected top-level keys exist before
       constructing the model, and/or
     - pass a JSON schema to the model (response_format json_schema / structured
       outputs) so the shape is enforced at the API level.

B. Add a "low-signal" guard in the outcome logic: if a file was analyzed but
   yielded zero comments AND zero positives AND zero recommendations, treat it as
   suspect (e.g. INCOMPLETE / COMMENT) rather than APPROVED, so a degenerate
   empty response can never auto-approve.

C. (Issue 1 robustness) wrap submit_review()'s create_review calls in
   try/except GithubException like assign_reviewers() already does, so a
   permissions problem downgrades gracefully instead of hard-failing the job.
